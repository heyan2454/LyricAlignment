## Superseding vowel-boundary decision

On 2026-07-22 the user accepted the vowel-boundary v2 grouping as the new M4Singer working contract, replacing the earlier 6,051 accepted / 14,845 review contract.  The v2 rule keeps consecutive identical vowel/final tokens within a held-vowel candidate, but starts a new candidate at either a new initial or a different vowel/final; a different vowel without an initial is retained as a zero-initial candidate rather than being absorbed into the preceding character.

The corresponding full rerun is external at `/home/hyan/Data/lyricalign/derived/20260722_m4singer_vowel_boundary_v2`: 20,896 total = 19,501 accepted + 1,395 review.  It records 13,450 review-to-accepted transitions versus the former grouping.  The review distribution is: multiple/complex slur with missing syllable groups 892; single slur with one missing syllable group 294; zero-initial or special-syllable case 173; multiple note groups per character 27; missing syllable groups 8; latin/digit 1.

Unresolved by explicit user direction: repeated finals in the `i**` and `u**` families, plus repeated `an`, `a`, `ou` and related finals, are intrinsically ambiguous under current metadata.  A repeated token can encode a held/slurred vowel or a separate lyric character.  No fixed allocation ratio, heuristic reassignment, or automatic acceptance change has been applied for this ambiguity.  A later review must use auditable evidence (for example manual samples, additional annotation provenance, or a separately validated rule) before refining it.

## Exact repeated-vowel allocation update

The user subsequently authorized one conservative automatic case: when an item has fewer candidate syllable groups than normalized lyric characters, count the repeated same-vowel tokens already held inside groups.  If and only if that count exactly equals the character/group deficit, split each repeated token into one zero-initial candidate group in source order.  This is an exact one-to-one allocation, not a percentage-based guess; surplus-group cases are not changed by this rule.

The v3 rerun at `/home/hyan/Data/lyricalign/derived/20260722_m4singer_repeated_vowel_exact_v3` replaces the preceding v2 working contract: 20,896 total = 19,709 accepted + 1,187 review.  It accepted 208 additional items with `accepted_rule_based_repeated_vowel_split`; 19,501 items remain accepted by the v2 vowel-boundary rule.  Remaining review is 1,186 slur/missing-syllable-group cases that do not satisfy exact allocation, plus 1 latin/digit record.  The broader ambiguity for repeated `i**`, `u**`, `an`, `a`, `ou`, and related finals remains unresolved whenever this exact condition does not hold.

## Unique pinyin validation update

All prior candidate mappings were subsequently checked against a multi-pronunciation Mandarin pinyin parse.  Heteronyms are retained; a mapping passes only where the full monotonic character-to-M4Singer-phoneme allocation is unique.  The parser may consume repeated identical finals as a hold, but rejects an item once more than one complete allocation exists.  Its reading conversion preserves zero-initial `y`/`w` orthographic syllables (for example `wu -> u`) before matching M4Singer tokens.  The corrected v4 output at `/home/hyan/Data/lyricalign/derived/20260722_m4singer_pinyin_validated_v4` is 20,896 total = 18,057 rule-validated + 2,839 review: 488 pinyin allocations are ambiguous and 2,351 have no exact phoneme match.  Future accepted manifests use `validation_basis: rule_validated` and `mapping_confidence: rule_validated`; they must not be described as high-confidence merely because they passed these rules.
